Clase 01 · Fundamentos

Introducción a la Ingeniería de Software

Qué es, por qué existe, y por qué código ≠ ingeniería de software.

¿Qué es la Ingeniería de Software?

Disciplina que aplica principios de ingeniería a la construcción de software. No se trata solo de escribir código: se trata de desarrollar sistemas de forma sistemática, comunicar áreas, dar visión estratégica y garantizar calidad.

La esencia de la Ingeniería de Software
  • Modelación: transformar el concepto en requerimientos, los requerimientos en diseño, el diseño en código fuente, y el código fuente en código ejecutable.
  • Diseño: encontrar la mejor solución que responda a los requisitos, sujeta a restricciones.
  • Optimización: encontrar las transformaciones de mejor calidad y más económicas.

Software Profesional: dos grandes categorías

TipoDescripciónQuién controla la especificación
Genérico (SaaS)Sistemas independientes producidos por una organización y vendidos en mercado abiertoLa organización que desarrolla
Personalizado (in-house)Destinado a un cliente particularLa organización que compra el software

Por qué la IS es relevante

Sistematización

Desarrollar ordenado, con visión estratégica y comunicación entre áreas.

Entender al usuario

Averiguar qué quiere y ofrecer soluciones reales.

Entrega de valor temprana

Liberar funcionalidades útiles lo antes posible.

Manejo de errores

Detectar y corregir problemas antes de que escalen.

Aseguramiento de calidad

Disminuir bugs y verificar que se cumplen las necesidades del cliente.

Metodologías

Marco para ejecutar el desarrollo de forma predecible.

Conceptos clave introducidos

  • Requerimientos: lo que el sistema debe hacer o satisfacer.
  • Producto: el software como entregable.
  • Delivery: la entrega al cliente/usuario final.
  • Time to market: cuánto tarda el producto en llegar al mercado.
  • MVP: Producto Mínimo Viable.

Dificultades para cumplir plazos

Estimación

Los desarrolladores tienden a subestimar el trabajo. Las estimaciones de esfuerzo son difíciles.

Corrección tardía

Corregir retrasos una vez que se producen es difícil.

Presión comercial

Stakeholders y área comercial presionan por plazos más cortos.

Trade-off

¿Qué pasa con la entrega de valor y el time to market?

Ética e Ingeniería de Software

La IS implica responsabilidades más amplias que las técnicas. Hay que comportarse de forma honesta y éticamente responsable. El comportamiento ético es más que cumplir la ley: implica seguir principios moralmente correctos.

Estándares éticos clave
  • Confidencialidad: respetar la de empleadores y clientes (aunque no se haya firmado acuerdo).
  • Competencia: no aceptar trabajos fuera de tu competencia ni desvirtuar tu nivel.
  • Derechos de propiedad intelectual: conocer leyes de patentes y copyright.
  • Mal uso de computadoras: no usar habilidades técnicas para dañar sistemas de otros (va desde lo trivial hasta difundir virus).

Códigos de referencia: Código de Ética y Conducta Profesional de la ACM, y Código de Ética Conjunto IEEE/ACM.

  • Diferencia entre producto genérico y personalizado (y quién controla la especificación).
  • Las 3 tareas esenciales de IS: modelación, diseño y optimización.
  • Conceptos clave: requerimientos, producto, delivery, time to market.
  • Responsabilidades éticas: confidencialidad, competencia, propiedad intelectual, mal uso.
Clase 02 · Modelos

Modelos y Procesos de Desarrollo I

Cascada, Incremental, Reutilización, Espiral y el Proceso Unificado Racional (RUP).

Proceso de Software

Un proceso de software es una serie de actividades relacionadas que conduce a la elaboración de un producto. Todo proceso debe incluir 4 actividades fundamentales:

  1. Especificación del software: funcionalidades requeridas y restricciones.
  2. Diseño e implementación: UX/UI, desarrollo (lenguaje), infraestructura.
  3. Validación: ¿cumple con lo que el cliente solicita?
  4. Evolución: el software evoluciona para seguir satisfaciendo necesidades.
Modelo de Proceso de Software Representación simplificada del proceso que ofrece información parcial. Los modelos se usan como marcos que se adaptan para crear procesos específicos.
Importante: no son mutuamente excluyentes — se combinan en la práctica.

Actividades del Proceso de Desarrollo

Estudios de factibilidad Levantamiento de requerimientos UX/UI Infraestructura Diseño Desarrollo (programación, testing, QA) Delivery Mejora continua Documentación Gestión de proyecto/producto

Modelo Cascada (Waterfall)

Ciclo de vida lineal. Cada etapa termina antes de iniciar la siguiente.

1. Definición de requerimientos 2. Diseño del sistema y del software 3. Implementación y prueba de unidad 4. Integración y prueba del sistema 5. Operación y mantenimiento // feedback flecha hacia arriba entre etapas adyacentes

Desarrollo Incremental

Actividades concurrentes (especificación, desarrollo, validación) que producen versiones: inicial → intermedias → final. Permite retroalimentación temprana y cambios durante el desarrollo.

Software Orientado a la Reutilización

Especificación de requerimientos → Análisis de componentes → Modificación de requerimientos → Diseño de sistema con reutilización → Desarrollo e integración → Validación.

Base para servicios web, objetos/paquetes y sistemas de software configurables.

Modelo en Espiral (Boehm)

No hay un número definido de iteraciones. Cada ciclo tiene 4 fases (enfocado en evaluación periódica del riesgo):

1. Planificación

Objetivos, alternativas, restricciones.

2. Análisis de Riesgos

Identificación, análisis y gestión.

3. Ingeniería

Desarrollo y validación del producto y su siguiente nivel.

4. Evaluación del cliente

Evaluación con stakeholders y planificación del siguiente ciclo.

Actividades del proceso de Ingeniería de Requerimientos

  1. Estudio de factibilidad → Informe de factibilidad
  2. Obtención y análisis de requerimientos → Modelos del sistema
  3. Especificación de requerimientos → Requerimientos del usuario y del sistema
  4. Validación de requerimientos → Documento de requerimientos

Actividades de Validación de Software

La V&V (verificación y validación) muestra que el sistema cumple con las especificaciones y expectativas del cliente.

  • Pruebas de desarrollo / unitarias: componentes individuales.
  • Pruebas de sistema / integración: componentes juntos.
  • Pruebas de aceptación: con el cliente / usuario final.

Proceso Unificado Racional (RUP)

Modelo híbrido (Kruchten, 2003). Se describe en 3 perspectivas:

  • Dinámica: fases del modelo a través del tiempo.
  • Estática: actividades del proceso.
  • Perspectiva: buenas prácticas utilizadas durante el proceso.
Fase RUPQué se haceEntregas
Concepción
(Inicio / Incepción)		 Objetivos, involucrados, visión (roadmap / user story mapping), requisitos de alto nivel (épicas), estimación preliminar, recursos, riesgos, equipo y skills Plan inicial, visión, lista de épicas
Elaboración Análisis de requerimientos, elaboración de arquitectura, análisis de riesgos, creación de prototipo inicial Documento de requerimientos, diagrama de arquitectura, prototipo inicial, plan de proyecto y entregables
Construcción Implementación, pruebas que garanticen requerimientos, pruebas de integración con otros sistemas, preparación para lanzamiento Código fuente, pruebas unitarias, manual de usuario, manual de instalación
Transición Entrega al cliente/usuario final, soporte y mantenimiento, evaluaciones y lecciones aprendidas Software terminado en producción, documentación de mantenimiento, plan de soporte

Mejores prácticas (RUP)

  • Desarrollo iterativo priorizado.
  • Gestión de requerimientos (documentación, tracking de cambios, análisis de impacto).
  • Arquitectura basada en componentes.
  • Software modelado visualmente (UML).
  • Verificación continua de calidad.
  • Control de cambios con sistema de administración.

Factores de Decisión del modelo

  • Importancia de especificación/diseño detallados antes de implementar.
  • Práctica de entrega incremental para obtener feedback.
  • Tamaño y tipo del sistema, tiempo de vida esperado.
  • Tecnologías disponibles, organización del equipo.
  • Problemas culturales, habilidades del equipo.

En la práctica: las empresas adoptan ágil para entregar valor e incrementar time to market, pero a veces se mezclan ágil con modelos basados en plan.

  • Las 4 actividades fundamentales de todo proceso de software.
  • Tipos de pruebas: unitarias (componentes), sistema (integración), aceptación (cliente).
Clase 03 · Ágil I

Metodologías Ágiles · Manifiesto · XP

De dónde viene lo ágil, por qué existe, y qué es Extreme Programming.

Características de las Metodologías Ágiles

  • Especificación, diseño e implementación mezclados (no secuenciales).
  • Desarrollo incremental por etapas.
  • Liberaciones / entrega de valor en cada ciclo.
  • Cada ciclo dura 2 a 3 semanas típicamente.
  • Se apoyan en herramientas específicas (testing, manejo de configs, integración, UI).

El Manifiesto Ágil

Declaración de valores y principios. Estamos descubriendo mejores formas de desarrollar software. A través de este trabajo hemos aprendido a valorar:

Los 4 valores ágiles
  • Individuos e interacciones sobre procesos y herramientas.
  • Software funcionando sobre documentación extensiva.
  • Colaboración con el cliente sobre negociación contractual.
  • Respuesta al cambio sobre seguir un plan.

Los 12 Principios (resumen operativo)

  • Máxima prioridad: satisfacer al cliente con entrega temprana y continua de valor.
  • Aceptar que los requisitos cambien incluso tarde (se aprovecha el cambio como ventaja competitiva).
  • Entregar software funcionando entre 2 semanas y 2 meses (preferencia por períodos cortos).
  • Negocio y desarrollo trabajan juntos a diario.
  • Individuos motivados, con el apoyo y entorno que necesitan.
  • Comunicación cara a cara como método más eficiente.
  • Software funcionando = principal medida de progreso.
  • Ritmo constante y sostenible.
  • Atención continua a la excelencia técnica y buen diseño.
  • Simplicidad (maximizar trabajo NO hecho).
  • Las mejores arquitecturas y requisitos emergen de equipos auto-organizados.
  • Reflexión periódica (retrospectivas).

Extreme Programming (XP)

Metodología ágil enfocada en mejorar la calidad del software y la capacidad de responder al cambio de requerimientos.

Comunicación

Entre desarrolladores, cliente y equipo.

Simplicidad

Diseñar para lo que se necesita ahora.

Retroalimentación

Constante del cliente y del propio código.

Coraje

Para refactorizar y tomar decisiones difíciles.

Respeto

Por el equipo y por el código.

Prácticas clave de XP

PrácticaQué es
IncrementalRequerimientos en Story Cards; las historias se eligen por tiempo disponible y prioridad. Spike: incremento sin programación (diseño arq. o documentación).
Liberaciones pequeñasEntregar versiones funcionales para retroalimentación temprana.
Diseño simpleSolo para la funcionalidad actual; no añadir más hasta que sea necesario.
Pruebas continuas (TDD)Framework de pruebas unitarias automatizadas. Se escriben las pruebas ANTES de implementar (test-first development).
RefactorizaciónTodos refactorizan continuamente para mantener el código simple y mantenible.
Programación en Pareja (Pair Programming)Dos desarrolladores trabajan juntos, uno comprueba el trabajo del otro.
Propiedad colectivaNo hay "islas de experiencia"; todos responsables de todo el código.
Integración continuaAl completar una tarea, se integra al sistema. Se generan pruebas de unidad.
Ritmo sustentableEvitar agotamiento del equipo.
Cliente en el sitio (on-site customer)El usuario de negocio es miembro del equipo XP y lleva los requerimientos.

Flujo XP

Seleccionar historia de usuario → Desglosar en tareas → Planear liberación → Desarrollar / integrar / probar → Liberación del software → Evaluar el sistema → (volver al inicio)

Pair Programming: ventajas vs desventajas

Ventajas
  • El sistema es de todos; todos responsables.
  • Parte del sistema de revisión.
  • Mejora la estructura del sistema.
  • Debug de issues complejos.
  • Entrenamiento de programadores junior.
Desventajas
  • No siempre necesario (otras formas de colaboración).
  • Puede ser más lento.
  • Un dev experimentado puede hacerlo solo en menos tiempo y costo.
  • Los 4 valores del Manifiesto Ágil (izquierda > derecha).
  • Los 5 principios clave de XP: comunicación, simplicidad, retroalimentación, coraje, respeto.
  • Qué es un Spike (incremento SIN programación).
  • TDD = test-first development: escribir pruebas ANTES del código.

SCRUM y Kanban

Los dos frameworks ágiles más utilizados. Roles, artefactos, ceremonias.

SCRUM · Conceptos Clave

ConceptoDefinición
SprintIteración o ciclo de software. Duración 2 a 4 semanas (1 para equipos expertos).
Product BacklogLista de funcionalidades a desarrollar para el producto, ordenadas por prioridad.
Sprint BacklogLista de funcionalidades/historias que se abordarán durante un sprint.
Velocity (velocidad)Estimado de cuánto puede hacer el equipo. Cantidad de historias entregadas del Sprint Backlog.

Roles en SCRUM

Dev Team

Grupo de desarrolladores auto-organizados.

Scrum Master

Responsable y guardián del Scrum. Coach del equipo; lo ayuda a alcanzar máxima productividad.

Product Owner

Responsable de maximizar el valor. Prioriza y ordena ítems del Product Backlog; entiende las historias.

Ceremonias SCRUM

CeremoniaCuándoPara qué
Sprint PlanningAl inicio del sprintPlanificar y estimar. Definir objetivo del sprint y comprometer historias.
Daily MeetingDiariaTodo el equipo revisa progreso y prioriza trabajo diario. Se declaran bloqueos, riesgos y necesidad de apoyo.
Sprint ReviewAl final del sprintPresentar trabajo completado. Retroalimentación de stakeholders, PO y asistentes.
RetrospectivaAl final del sprintEl equipo reflexiona y encuentra puntos de mejora para el siguiente sprint.

Flujo SCRUM

Product Backlog (priorizado) ↓ Sprint Planning Sprint Backlog (subset comprometido) ↓ Sprint de 2-4 semanas │ ↻ Daily Meetings Incremento terminado → Sprint Review → Retrospectiva ↓ Siguiente Sprint

Kanban

Metodología de gestión de proyectos que optimiza el desarrollo mediante flujo continuo de trabajo.

Kanban significa "tarjeta de señal": cada unidad de trabajo se mueve por el flujo solo cuando hay capacidad en el siguiente paso.

Scrum vs Kanban
  • Scrum trabaja por sprints con compromisos fijos.
  • Kanban se enfoca en limitar el trabajo en progreso (WIP) en cada etapa, permitiendo entregas frecuentes y ajustes continuos.
  • Ambos son incrementales: dividen el trabajo en partes (historias, tareas).

Los 5 Principios de Kanban

#PrincipioQué implica
1Visualizar el trabajoTodos ven el estado en cualquier momento. Promueve transparencia y comunicación.
2Limitar el WIPNúmero máximo de tareas en cada etapa. Previene sobrecarga y minimiza tiempo de ciclo.
3Gestionar el flujoOptimizar el trabajo a través del progreso. Identificación de bloqueos.
4Políticas explícitasCriterios claros para que una tarea pase a la siguiente columna.
5Colaboración / evolución / experimentaciónMejora continua del proceso y el producto.

Tablero Kanban típico

│ BACKLOG │ STORIES │ TO DO │ IN PROGRESS │ TO VERIFY │ DONE │

Cada columna puede tener un WIP limit. Las tarjetas fluyen de izquierda a derecha según capacidad.

  • Duración típica de un Sprint: 2 a 4 semanas.
  • Los 3 roles SCRUM y quién prioriza el backlog (el Product Owner).
  • Las 4 ceremonias SCRUM y el propósito de cada una.
  • WIP limit: concepto central de Kanban (no de Scrum).
  • Kanban = flujo continuo. Scrum = iteraciones con compromiso.
Clase 05 · Requerimientos

Requerimientos Funcionales y No Funcionales

FR, NFR, historias de usuario INVEST, criterios de aceptación y el documento SRS.

Introducción a los Requerimientos

  • Existen dos clases: de usuario (funcionales) y de sistema (no funcionales).
  • Establecen necesidades y limitaciones del cliente.
  • Se escriben con distintos niveles de detalle según la audiencia.
  • Un documento de requerimientos puede ser parte de un contrato.

Requerimientos Funcionales (FR)

Requerimiento Funcional Enunciados sobre los servicios que el sistema debe proveer, cómo debe reaccionar a entradas y comportarse.
  • Lectores: gerentes de negocio, usuarios finales, ingenieros/analistas, arquitectos, contratistas.
  • Escritos en lenguaje natural a alto nivel para que el cliente lo entienda.
  • Pueden incluir diagramas de servicio.
  • Suelen venir apalancados por un proceso de negocio.
  • Al ser de alto nivel pueden ser ambiguos o de alcance no acotado.
    Ejemplo: "Un usuario debe ser capaz de buscar las listas de citas de todas las clínicas".

Historias de Usuario (US) en Desarrollo Ágil

En ágil, los FR se expresan como historias de usuario. Forman parte de un Product Roadmap y nutren el Product Backlog.

Como: "Rol" Quiero: "funcionalidad" Para: "beneficio" Supuesto: "relación / condición aceptada sin prueba" Criterios de Aceptación: 1. Criterio 1 2. Criterio 2 3. Criterio 3

Supuesto: creencias que el equipo asume como verdaderas sin pruebas inmediatas. Ejemplo: "El sistema siempre tendrá conexión a internet" o "El usuario ya ha iniciado sesión".

INVEST · criterios de una buena historia

I · Independent

Independiente de otras historias.

N · Negotiable

Negociable en su alcance.

V · Valuable

Entrega valor al usuario / negocio.

E · Estimable

Se puede estimar su esfuerzo.

S · Small

Pequeña: cabe en un sprint.

T · Testable

Verificable objetivamente.

Criterios de Aceptación

La historia define una dirección, NO la implementación. Los criterios convierten la intención en condiciones claras y comprobables que determinan cuándo la historia está terminada, alinean al equipo, eliminan ambigüedades y proporcionan un estándar medible para QA y stakeholders.

Ejemplo

Como cliente, quiero buscar productos por nombre para poder encontrar fácilmente lo que busco.

Criterios de Aceptación:

  • La función devuelve resultados que coinciden exactamente con el nombre ingresado.
  • La función devuelve resultados que coinciden parcialmente con el nombre ingresado.
  • Los resultados se muestran en un formato claro, organizado y fácil de usar.

Buenos criterios de aceptación son...

AtributoQué significa
Claros y concisosLenguaje sin jerga técnica ni ambigüedades.
TesteablesVerificable objetivamente; mapean a pruebas ejecutables.
Resultado, no recetaDescribir qué experimenta el usuario, no los pasos técnicos.
MediblesCuantificar. Reemplazar "debería verse bien" por "imagen de mínimo 300×300 px".
IndependientesCada criterio autónomo; simplifica pruebas y diagnóstico.

Requerimientos No Funcionales (NFR)

Requerimiento No Funcional Limitaciones a los servicios/funciones del sistema. Definen propiedades (tiempo de respuesta, almacenamiento, etc.). Suelen ser más críticos y restrictivos que los FR, e incluso pueden restringir FR existentes.
  • Se aplican al sistema completo o a servicios individuales.
  • Tienen métricas: son medibles y concretos (aplica / no aplica).
  • Ejemplos: SO, versión de navegador, versión de librería, lenguaje de programación.

Clasificación de NFR

CategoríaEjemplos
Del ProductoVelocidad de ejecución, fiabilidad de datos, % de error aceptado. Incluye eficiencia, rendimiento, espacio, usabilidad, confiabilidad, seguridad.
De la OrganizaciónPolíticas y procedimientos organizacionales, estándares, requisitos de implementación. Incluye ambientales, operacionales, de desarrollo.
ExternosExternos al sistema: proceso de desarrollo, interoperabilidad, legislativos. Incluye regulatorios, éticos, legales, contables, de protección/seguridad.

Estos requerimientos forman parte de un contrato.

Métricas típicas de NFR

PropiedadMedida
VelocidadTransacciones/segundo · tiempo de respuesta · tiempo de refresco de pantalla
TamañoMBytes · número de ROM chips
Facilidad de usoTiempo de formación · nº de fotogramas de ayuda
ReliabilityTiempo medio de fallo · probabilidad de indisponibilidad · disponibilidad
ConfiabilidadTiempo para reiniciar · % de eventos causando insuficiencia · probabilidad de corrupción de datos
Portabilidad% de estados dependientes del destino · nº de sistemas de destino
Ejemplos de texto (MHC-PMS)
  • Producto: "El MHC-PMS estará disponible en todas las clínicas durante horas de trabajo normales. Los tiempos muertos no rebasarán los 5 segundos."
  • Organizacional: "Los usuarios se acreditarán con la tarjeta de identidad de la autoridad sanitaria."
  • Externo: "Como establece la HStan-03-2006-priv, el sistema implementará provisiones para la privacidad del paciente."

Documento SRS (Software Requirements Specification)

Comunicado oficial de lo que deben implementar los desarrolladores. Incluye requerimientos de usuario y especificación detallada del sistema. Esencial cuando un contratista externo diseña el sistema. Estándar IEEE 830.

Secciones típicas del SRS

  • Prefacio (versiones, cambios, lectores esperados)
  • Introducción (necesidad, encaje con otros sistemas, objetivos)
  • Glosario
  • Definición de requerimientos del usuario
  • Arquitectura del sistema
  • Especificación de requerimientos del sistema (FR y NFR detallados)
  • Modelos del sistema (UML, flujos, objetos)
  • Evolución del sistema
  • Apéndices / Índice

Formas de notación

  • Enunciados en lenguaje natural numerados.
  • Lenguaje natural estructurado (plantillas).
  • Lenguajes de descripción de diseño.
  • Anotaciones gráficas (UML, diagramas de secuencia, casos de uso).
  • Especificaciones matemáticas (poco usadas: los clientes no las entienden).
  • Diferencia FR vs NFR: FR = servicios; NFR = propiedades/limitaciones.
  • El acrónimo INVEST completo.
  • Las 3 categorías de NFR: Producto / Organización / Externos.
  • Partes de una historia de usuario: Como / Quiero / Para + supuestos + criterios.
  • Los NFR pueden restringir a los FR, no al revés.
Clase 06 · Requerimientos II

Levantamiento de Requerimientos

El proceso de descubrir lo que el cliente realmente necesita (y las técnicas para hacerlo).

Proceso de Levantamiento de Requerimientos

  1. Descubrimiento: interacción con stakeholders para descubrir necesidades (funcionales, no funcionales y de dominio).
  2. Clasificación y organización: se agrupan en módulos, procesos, comunicación entre sistemas.
  3. Priorización y negociación: se prioriza y se abordan conflictos.
  4. Especificación: se documentan y generan versiones. Se itera.

Requerimientos de Dominio

Requerimiento de Dominio Deriva del dominio de aplicación. Pueden ser FR nuevos, restricciones funcionales o cálculos particulares.
  • El problema: es difícil detectar cuándo un requerimiento de dominio se pierde o entra en conflicto con otros.
  • Ejemplos:
    • Sistema de control de trenes → características de frenado en distintas condiciones climáticas.
    • Sistema de control de vuelos → características de la pista, vientos, condiciones climáticas extremas.
  • Si no se satisfacen, el sistema queda inutilizable.

Técnicas de Levantamiento

TécnicaCaracterísticas / Cuándo usarla
Entrevistas Formales o informales, con usuarios clave o stakeholders. Pueden ser estructuradas (guion fijo) o semi-estructuradas (con flexibilidad). Difícil capturar requerimientos de dominio (jerga, puntos ciegos).
Escenarios y Casos de Uso Descripción completa del flujo. Debe incluir: (1) qué esperan sistema y usuarios al inicio, (2) flujo normal de eventos, (3) qué puede salir mal y cómo manejarlo, (4) actividades en paralelo, (5) estado al terminar.
Levantamiento del proceso de negocio Cómo funciona hoy y cómo puede ser más eficiente. De alto nivel → más detalle. Identifica usuarios clave por etapa.
Focus groups / Workshops Dinámica grupal para alinear necesidades entre varios stakeholders.
Encuestas Recolección cuantitativa de feedback amplio.
Prototipado Construir versiones rápidas para validar ideas visual e interactivamente.

Clase 07 · Producto

Roadmaps y User Story Mapping

De la visión de producto a la planificación visual de historias.

Visión de Producto (Product Vision)

[Nombre del Producto] atiende a [Cliente/Usuario Objetivo], quienes buscan [Necesidades] para [Beneficio del Producto]. Nuestra visión es [cómo el producto cambiará la vida de los clientes en una ventana de tiempo]. Lo que nos diferencia de [Competidor Principal] es que nuestro producto [Ventaja Competitiva Clave].

Roadmap de Producto

Roadmap de Producto Mapa de las principales funcionalidades de un producto. Es la intención de lo que se desea desarrollar. Contiene visión y objetivos (organizacionales y del producto). Tiene hitos visibles, etapas y, a veces, quarters esperados.
  • No es una Gantt.
  • En muchas compañías se usa para establecer metas claras con cada hito (sujeto a presupuesto).
  • Se presentan o crean en una ceremonia llamada "Inception" (Agile Inception = Kick Off; se comparte y concreta la visión del producto).

Dimensiones de un Roadmap

DimensiónDescripción
Fecha / VersiónPeríodo en el que el objetivo debe lograrse.
NombreNombre de la nueva versión (ej. "Bronze", "Silver", "Gold", "Platinum").
ObjetivoResultados o beneficios a alcanzar.
FuncionalidadesFuncionalidades de alto nivel requeridas para cumplir el objetivo.
MétricasMediciones para determinar si el objetivo se logró.

Preguntas clave al construirlo

  • ¿Cuál es la brecha entre los resultados que los clientes experimentan hoy y los que les gustaría?
  • ¿Cuáles son los próximos objetivos que podrían cerrar esas brechas?
  • ¿Qué experimentos ayudarían a validar nuestras suposiciones?
Importante
  • Actualizarlo periódicamente (en refinamientos del backlog o Sprint Reviews).
  • Mantenerlo a alto nivel y ligero.
  • Enfocarse en objetivos, no detalles. Da flexibilidad y creatividad.

Mapa de Historias de Usuario (User Story Mapping)

Visión de las historias de usuario que serán abordadas en cada etapa del roadmap. Flexible (puede cambiar).

Aspectos al elaborarlo

  • Línea de tiempo de izquierda a derecha.
  • Priorización arriba hacia abajo (más prioritarias arriba).
  • Agrupar las historias en grupos de alto nivel (épicas e iniciativas).
  • Puede planificarse en versiones de producto.
  • Permite visualizar claramente el MVP.
  • Permite visualización e interacción de roles.

Jerarquía: Iniciativa → Épica → Historia → Tarea

CaracterísticaIniciativaÉpicaHistoria de Usuario
TamañoMuy GrandeGrandePequeña / Funcional
DuraciónMeses / TrimestresVarios SprintsUn Sprint o menos
EnfoqueEstratégico (Qué / Por qué)Funcionalidad / ProductoTécnico / Accionable (Cómo)

Ejemplo concreto

Iniciativa: Mejorar el proceso de pago.

Épica: Implementar pago con tarjeta de crédito.

Historia de Usuario: Como usuario, quiero guardar mis datos de tarjeta para no volver a escribirlos.

Elaboración paso a paso de un User Story Mapping

  1. Identificar usuarios, roles y objetivos (enfocado en necesidades).
  2. Definir el Journey del Usuario (Customer Journey Map).
  3. Desglose de user stories (funcionales y no funcionales).
  4. Priorización vertical: las más prioritarias arriba.
  5. Añadir detalles, subtareas y dependencias.
  6. Criterios de aceptación.
  7. Acuerdos: DoR (Definition of Ready) y DoD (Definition of Done).
  8. Identificar releases o versiones.
  9. Revisar y ajustar en cada iteración.

Definition of Done (DoD) · ejemplo

Una historia está "Done" cuando...
  • Todo el código fuente está completo y actualizado en el repositorio.
  • Se han escrito y aprobado pruebas unitarias para cada función; se ejecutan satisfactoriamente.
  • No existen errores de integración.
  • Se realizaron pruebas de aceptación con el cliente / usuario final.
  • El Product Owner está conforme con el incremento.
  • Documentación completa y actualizada (usuario, desarrollo, ciberseguridad).
  • El roadmap NO es una Gantt.
  • Las 5 dimensiones de un roadmap: fecha, nombre, objetivo, funcionalidades, métricas.
  • Jerarquía: Iniciativa > Épica > Historia > Tarea.
  • En USM: tiempo → izquierda a derecha; prioridad → arriba hacia abajo.
  • Diferencia DoR vs DoD: DoR = lista para iniciar; DoD = completada.
Clase 08 · Planificación

Priorización y Estimación Ágil

Cómo ordenar el backlog y estimar historias.

Product Backlog y Planificación

  • Listado de funcionalidades a desarrollar para el producto.
  • Deben ser priorizadas en lista descendente (más prioritario arriba).
  • En la ceremonia de planificación se revisa el product backlog.
  • Se establece el objetivo y entrega de valor.
  • Se analiza cada historia: tasking, reglas del equipo, DoR y DoD.
  • El equipo se compromete con un subgrupo de historias para el Sprint.

Priorización: Característica vs Necesidad

Regla práctica típica (distribución que aparece en la clase):

Característica% ProductoNecesidad real
Críticas20%80%
(lo que realmente se necesita)
+/- Importantes40%
(secundarias)20%
Triviales / deseables (no impactan el ahora)20%20%

Planificación y Capacidad del Equipo

  • La capacidad = cantidad de historias que pueden desarrollarse en un sprint.
  • Si el equipo es nuevo y no conoce su capacidad:
    • Definir un baseline (ej. 8 puntos por persona).
    • Si no se logra el objetivo → analizar en retrospectiva cómo mejorar.
    • Resetear la capacidad del equipo con esos datos.
    • Los datos importan: métricas, cambio de estrategia, tareas de mejora.

Gestión del Sprint Backlog

  • El equipo elige cómo abordar las historias del sprint backlog.
  • Las historias NO se asignan: cada miembro toma una historia de la pila.
  • Las tareas se actualizan a medida que avanza el sprint (bloqueos, riesgos, apoyo → se declaran en el Daily).
  • El objetivo del sprint NO se modifica.
  • Cualquier cambio debe ser acordado por el equipo junto con el Product Owner.
Burn Down Chart

Gráfico que muestra esfuerzo remanente vs remanente ideal a lo largo del sprint. Permite ver si el equipo está avanzando según lo planificado.

Estimación Ágil

Existen distintas técnicas. Dos ejes comunes:

  • Esfuerzo vs duración
  • Valor vs esfuerzo

La estimación la realiza el equipo de desarrollo en el Sprint Planning. Hay estándares de duración máxima de la planificación. Para una buena estimación es necesario establecer una Historia Pivote.

Técnica 1 · Tamaño Camiseta (T-Shirt Sizing)

Se asigna talla XS, S, M, L, XL, XXL, XXXL a cada historia según complejidad.

  1. Pivote: tiende a ser XS o S.
  2. El equipo establece criterios que cumple la historia pivote.
  3. Luego criterios para las siguientes tallas.
  4. Proceso en 3 etapas: estimación individual → grupal (tendencia) → ajuste y consenso.

Técnica 2 · Fibonacci (la más usada)

Secuencia: 1, 2, 3, 5, 8, 13, 20, 40, 100, ?, ∞, ☕ (pausa)
  • Se establece un pivote (tiende a ser 1 o 2).
  • Herramienta típica: Planning Poker (cartas).
  • Mismo flujo: estimación individual → grupal → ajuste.
  • Las cartas "?" y "Café" indican "no tengo claridad" o "pausa".

Flujo Planning Poker

Join Session → Discussing the User Story → Secret Selection of Cards → Displaying All Cards at Once → Discuss Highest and Lowest Estimates → Consensus? ───yes──▶ Success └──no──▶ (volver a discutir)
Trampa común

"Usar Fibonacci como tallas de camiseta" NO es recomendado (confunde los sistemas). T-shirt y Fibonacci son técnicas distintas.

  • Las historias en Scrum NO se asignan: se toman de la pila.
  • El objetivo del sprint NO se modifica durante el sprint.
  • La estimación la hace el equipo en el Sprint Planning.
  • Secuencia Fibonacci: 1, 2, 3, 5, 8, 13, 20...
  • Las 3 etapas de estimación: individual → grupal → ajuste.
  • La historia pivote suele tener valor 1 o 2 / talla XS o S.
Clase 09 · Arquitectura

Arquitectura de Software

Cómo organizar un sistema en componentes y los patrones más importantes.

Qué es la Arquitectura de Software

Arquitectura de Software Se preocupa de entender cómo un sistema debe organizarse y diseñar su estructura. Es el primer paso del diseño. Es la relación vital entre ingeniería de requerimientos y diseño: identifica los componentes principales y sus relaciones.
  • Estructura: descomposición en componentes y sus interacciones.
  • Grandes decisiones de diseño basadas en requerimientos (incluyendo los NO funcionales).
  • En metodología ágil: se hace un diseño arquitectónico inicial. Luego no se modifica en completitud sino que va escalando (por el costo).

Arquitectura vs Diseño

La separación no es nítida, pero son claramente de arquitectura:

  • La forma del sistema: client-server, web-based, native mobile, desktop app, etc.
  • Estructura del software (componentes, capas).
  • Tecnologías (lenguaje, plataforma de deployment).
  • Frameworks.

La arquitectura es principalmente descomponer el sistema en componentes.

¿Por qué es importante?

Afecta el rendimiento, robustez, distribuibilidad y mantenibilidad. Los componentes individuales describen características funcionales; la arquitectura describe las no funcionales.

Comunicación

Entre los stakeholders.

Análisis

Reutilización

A gran escala.

Cómo describir una arquitectura

  • Mediante diagramas (generalmente simples, de alto nivel).
  • Se puede usar UML (despliegue, componentes).
  • Modelo C4 (Simon Brown): 4 niveles jerárquicos — Contexto, Contenedores, Componentes, Código.
  • Diagrama de bloque: simple, entendible por todos los stakeholders, permite tomar decisiones de alto nivel.

Patrones o Estilos Arquitectónicos

Patrón / Estilo Arquitectónico Descripción de la organización del sistema (por ej. cliente-servidor, en capas). Capturan la esencia de una arquitectura usada en distintos sistemas.

1 · Cliente-Servidor

Funcionalidad organizada en servicios, cada uno entregado por un servidor independiente. Los clientes ingresan a los servidores para usarlos.

Ejemplos: aplicaciones web y móviles, sistemas de bases de datos, correo electrónico.

Ventajas
  • Facilita distribución y escalabilidad.
  • Clara separación de responsabilidades.
Desventajas
  • Cuellos de botella en el servidor.
  • Problemas de seguridad y autenticación.

2 · Filtros y Tuberías (Pipe-and-Filter)

Los componentes (filtros) procesan datos y los pasan a través de conectores (tuberías).

Ejemplos: compiladores, procesamiento de imagen, pipelines de datos.

Ventajas
  • Modularidad y reutilización.
  • Fácil de entender y adaptar.
Desventajas
  • Mayor latencia entre filtros.
  • Menos adecuado para aplicaciones interactivas.

3 · Modelo-Vista-Controlador (MVC)

Separa presentación e interacción con los datos. Tres componentes lógicos:

  • Modelo: maneja los datos y la lógica de negocio.
  • Vista: define y maneja cómo se presentan los datos al usuario.
  • Controlador: maneja la interacción del usuario.

Ejemplos: aplicaciones web y móviles; interfaces de usuario.

Ventajas
  • Datos cambian independiente de su representación (y viceversa).
  • Soporta presentar los datos de distintas maneras.
Desventajas
  • Costos adicionales / complejidad cuando el modelo e interacciones son simples.

4 · Arquitectura de Capas (Layered)

Organiza el sistema en capas con funcionalidad relacionada. Una capa da servicios a la capa de encima. Las capas inferiores representan servicios núcleo. Una capa es una colección coherente de componentes.

Interfaz de usuario Gestión de interfaz · Autenticación/autorización Lógica empresarial · Funcionalidad de aplicación · Utilidades Soporte del sistema (OS, base de datos, etc.)

Permite desarrollo incremental por capas. Las capas pueden cambiarse; al hacerlo solo se modifica la capa inferior.

Ejemplos: aplicaciones web y empresariales; sistemas operativos.

Ventajas
  • Separación de responsabilidades.
  • Reutilización de componentes.
Desventajas
  • Latencia por comunicación entre capas.
  • Comunicación compleja a medida que crece el sistema.

5 · Repositorio / Memoria Compartida

Todos los datos se gestionan en un repositorio central accesible a todos los componentes. Los componentes NO interactúan directamente, solo a través del repositorio.

Se privilegia su uso cuando hay componentes que modifican información que otros usan.

Ejemplos: sistemas de control de versiones, sistemas de gestión de contenido (CMS).

Ventajas
  • Gestión y control de datos.
  • Reutilización e interoperabilidad.
Desventajas
  • Cuellos de botella y problemas de escalabilidad.
  • Difícil de mantener.

6 · Arquitectura Orientada a Servicios (SOA)

Interacción entre servicios autónomos que realizan funciones específicas. Componentes débilmente acoplados.

  • Un servicio es un componente que no corre en el mismo proceso y se comunica por protocolo (como HTTP).
  • Proveen funcionalidades según una especificación de interfaz.
  • Pueden combinarse dinámicamente.
  • Íntimamente ligado a Web Services y formalización de procesos de negocio.

Ejemplos: sistemas de gestión empresarial (ERP).

7 · Microservicios

Una aplicación se construye con un conjunto de pequeños servicios (HTTP, REST APIs). Los servicios pueden escribirse en distintos lenguajes y usar distintos almacenamientos.

  • Rompe en pequeños servicios la componente que usualmente corresponde al server.
  • Altamente cohesivos y desacoplados.
  • Desarrollo, deployment, gobierno y datos descentralizados (cada servicio es totalmente independiente).
  • Surgen 2011-2012. Algunos los ven como "SOA hecho bien" (SOA se había vuelto demasiado complejo).
  • No requieren capa de middleware: cada servicio expone una API (contrato).
  • Servicios principalmente funcionales, con algunos de infraestructura (no expuestos).

Ejemplos: Netflix, Amazon, apps en la nube.

Microservicios y Contenedores

  • Necesidad de manejar cientos de microservicios independientes.
  • Un microservicio puede correr en su máquina virtual, pero hoy es popular la tecnología de contenedores.
  • Un contenedor provee todo lo necesario para el servicio, siendo mucho más liviano que una VM (no incluye un SO completo).
  • Docker es la plataforma más popular. No es la única forma de desplegar microservicios.

Errores que suelen entrar en preguntas

No te equivoques
  • Roadmap ≠ Gantt.
  • En Scrum, las historias no se asignan: cada dev toma una de la pila.
  • El objetivo del sprint no se modifica una vez iniciado.
  • Fibonacci y T-shirt son distintas técnicas: mezclarlas no se recomienda.
  • Los NFR pueden restringir a los FR (no al revés).
  • En XP, TDD es test-first: las pruebas se escriben ANTES del código.
  • Un Spike es un incremento SIN programación.
  • MVC: el Modelo tiene la lógica de negocio, NO la vista.
  • Microservicios no necesitan capa de middleware; SOA sí la puede tener.
  • Kanban usa WIP limit; Scrum NO.
